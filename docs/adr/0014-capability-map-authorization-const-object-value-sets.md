# 0014. Capability-map authorization and const-object value sets

- **Status:** Accepted
- **Date:** 2026-07-11
- **Related:** ADR-0009 (admin gate); ADR-0013 (thin actions/domain-organized lib — structural predecessor); technical-design.md §5.7, §6; PR review (admin-authoring epic)

## Context

PR review on the admin-authoring epic flagged two recurring problems:

1. **Scattered role checks.** Authorization was expressed five different ways across the codebase — a bare `role === "admin"` here, a `role === "author" || role === "admin"` there, ad hoc `isStaff`/`isAdmin` booleans threaded through props, and a `staffSession`/`adminSession` pair of guard functions that each re-derived the same role logic independently. Adding a new capability (or a new role) meant grepping for every shape and updating each one by hand; a review pass could easily miss a call site, silently under- or over-granting access.
2. **String literals for bounded value sets.** `PostStatus`, `Role`, and the admin dashboard's sort key were plain string literals compared inline (`status === "published"`), with no single place enumerating the valid set or its pg-enum correspondence. That invited typos that TypeScript couldn't catch and made "what are all the statuses" a grep exercise instead of a type.

Both problems share a root cause: no single source of truth for a small, closed set of values (roles, actions, statuses) that the rest of the codebase should treat as opaque tokens rather than re-deriving.

## Decision

- **One capability layer:** `src/lib/auth/permissions.ts` defines an `Action` const object (`post.create`, `post.publish`, `admin.access`, …) and a `ROLE_ACTIONS: Record<Role, readonly Action[]>` map, exposed through a single function, `canPerformAction(user, Action.X)`. Every gate — server action, page/layout guard, UI conditional (e.g. `header-auth.tsx`) — calls this function instead of comparing `role` inline. `staffSession`/`adminSession`/`isStaff`/`isAdmin` are removed; server actions gate through `actionSession(action)` (`src/lib/auth/guards.ts`), which wraps `getSession()` + `canPerformAction()` into one call. Ownership scoping (author limited to `author_id = self`) stays a separate, explicit check next to the capability check — it isn't a capability, and `Action.ManageAnyPost` is the bypass a service checks to decide whether to apply that scope.
- **`permissions.ts` is client-safe** (no `server-only`, no schema import) so `header-auth.tsx` can gate UI without a server round trip. This intentionally decouples it from the `user_role` pg enum; the drift guard between `Role` and the enum lives instead in `src/lib/auth/roles.test.ts`, which is the one place that needs the schema import.
- **Bounded value sets get one canonical declaration, not the TS `enum` keyword.** `PostStatus` (`src/lib/posts/status.ts`) and `Role` (`src/lib/auth/roles.ts`) are `const` objects + `as const` with a derived literal-union type of the same name, each with a `_VALUES`/`_STATUSES` array asserted against its pg enum in a test; `ADMIN_POST_SORTS` (`src/lib/posts/admin.ts`), which has no pg-enum counterpart and no need for named keys, is a `const` tuple array + `as const` with the union type derived via `(typeof ADMIN_POST_SORTS)[number]`. Both shapes share the point of the rule: one declaration, a derived type, no restating the set elsewhere. `enum` is rejected outright — it has a runtime emit that a plain object/array with `as const` doesn't need, doesn't tree-shake as well, and doesn't work with `erasableSyntaxOnly`. Non-test code never compares a raw literal (`"published"`, `"admin"`) against these fields; it goes through the named constant (`PostStatus.Published`, `Role.Admin`) or, for authorization, through `canPerformAction`. Tests are the sanctioned exception — asserting a raw literal there pins the external contract (the actual string stored in Postgres / sent to the client) so an accidental rename of the constant's _value_ still fails a test, not just a type check. A single module's own internal discriminated-union tags (e.g. `data.ts`'s `{ ok: false; reason: "slug-collision" }`) aren't a cross-cutting value set and can stay inline literals.

## Consequences

- Easier: a new role or a new capability is a one-map edit (`ROLE_ACTIONS`) instead of a hunt across five call-site shapes; UI gating and server gating share the exact same predicate, so they can't drift; `rolesWithAction(action)` derives "which roles can do X" from the same map for UI that needs to list staff (the admin dashboard's author filter), removing a second list that had to stay in sync by hand; a bounded value set is grep-able and type-checked from one file, and a typo'd literal is a compile error rather than a silent no-match.
- Harder: adding a capability requires touching the shared `permissions.ts` map even for a single call site, which is marginally more ceremony than an inline check for a genuinely one-off gate; keeping `permissions.ts` client-safe means the `Role`/pg-enum drift guard has to live in a second file (`roles.test.ts`) rather than colocated with the map itself, which is one extra place to know about when adding a role.
- Complements ADR-0013: that ADR gave each domain a `service`/`data` split and thin action boundaries; this ADR gives the codebase one shared authorization vocabulary (`Action`, `canPerformAction`) and one pattern for closed value sets, both used consistently across those service/data/action layers.
- `.claude/rules/engineering.md` is updated with these as standing rules (capability-map gating, ownership as a separate check, const-object-over-enum value sets, comments-explain-why, exhaustive-lib-tests-vs-representative-action-tests) so they're enforced on every future change. `technical-design.md` §5.7 already described the conceptual "session → role → ownership" check without naming the old symbols, so no deviation-from-locked-doc language was needed there; §6's `lib/auth/` sketch already listed `permissions.ts` and needed no edit.
